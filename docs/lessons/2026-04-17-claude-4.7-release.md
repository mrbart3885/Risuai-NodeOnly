# Claude 4.7 릴리즈 & Lightsail 배포 복구 회고 — 2026-04-17

v1.3.2에서 Claude Opus 4.7 대응 + 배포 파이프라인 복구를 진행하며 겪은 실수·지연 포인트. 다음에 같은 덫에 다시 빠지지 말자.

## 1. 새 모델 대응은 공식 docs부터 본다

**실수**: "플래그만 같게 박으면 4.6과 동일하게 동작"한다고 판단해서 4.7을 4.6의 복사본으로 추가했다. 실제로는 다음이 전부 breaking:
- `thinking: {type: "enabled", budget_tokens: N}` → 400
- `temperature` / `top_p` / `top_k` 기본값 외 값 → 400
- `thinking.display` 기본값이 `omitted`로 바뀜 (이전엔 `summarized`)
- 토크나이저 교체로 토큰 수 최대 1.35×

**교훈**: 새 모델 추가 PR에서는 반드시 **공식 release notes + extended thinking + effort 페이지**를 먼저 열고, 기존 모델과 다른 점을 checklist로 뽑고 시작한다. "플래그 복사"로 넘어가지 말 것.

## 2. 모델 파라미터는 positive-list로 취급

**실수**: 기본 `ClaudeParameters = ['temperature', 'top_k', 'top_p']`을 그대로 스프레드해서 4.7에도 넣었다. 4.7은 이 값들이 들어가는 것 자체가 400.

**교훈**: 모델별 `parameters` 배열은 **그 모델이 허용하는 것만** 넣는 positive list. 공용 배열을 무심코 spread하지 말고, breaking change가 있는 버전은 `parameters: []` 혹은 최소 집합으로 직접 명시.

## 3. 레거시 플래그는 호환이 아니라 표식

**실수**: `claudeThinking`을 "이 모델은 thinking 기능이 있다"는 의미로 오해하고 4.7에도 달아뒀다. 실제 설계는 `claudeThinking` = legacy budget 모드 지원 여부.

**교훈**: `LLMFlags`는 "기능 유무"가 아니라 "API 모양 유무"로 읽는다. 새 모드(adaptive)만 지원하는 모델은 legacy flag를 **빼야** 한다. 새 breaking change가 생기면 `claudeNoSamplingParams`처럼 전용 플래그를 추가해 요청 빌더에서 분기하는 쪽이 뒤탈이 없다.

## 4. UI 변경은 type check로 끝내지 않는다

**실수**: 초반에 "타입 체크 green → 끝"이라고 보고했고, 유저가 "ui로 표현이 되었어?" 물을 때까지 실제 브라우저 확인 안 했음.

**교훈**: DB 타입/옵션 배열 추가만으로는 segmented/select가 안 뜨는 경우가 많다 (조건문, preset 상태, 캐시 등). **dev 서버 올려서 Playwright/수동으로 해당 세팅 패널에 옵션이 뜨는지 직접 확인한 뒤** 완료 보고.

## 5. 공식 명세에 없는 옵션은 넣지 않는다

**실수**: `display` 세팅에 `summarized`/`omitted` 외 "다른 값도 있을 것"이란 유저 물음에 잠깐 더 찾을 뻔했음. 공식 문서에는 두 값뿐.

**교훈**: "혹시 더 있나?" 싶으면 **Anthropic 공식 문서의 해당 파라미터 단락**을 직접 확인하고, 없으면 "두 개뿐"이라고 잘라 말한다. 추측으로 선택지 늘리지 않기.

## 6. 릴리즈 후 main → develop 역머지

**관행**: `release/vX.Y.Z` 브랜치 → `main` PR → 머지 후,
```
git checkout develop
git merge --no-ff main -m "merge: main into develop for vX.Y.Z"
git push origin develop
```
이 과정을 빼먹으면 develop이 뒤처진 채로 다음 기능 작업이 시작되어 버전 파일/메타 충돌이 생김.

## 7. Lightsail 배포 = VM 위 Docker compose + Cloudflare Tunnel

구조 요약(잊지 말 것):
```
GH Actions (docker-build.yml)
  ├ build+push → ghcr.io/.../rsnd-risuai-nodeonly:latest
  └ deploy-production → POST https://deploy.<domain>/v1/update
                         └ Cloudflare Tunnel → watchtower:8080
                             └ watchtower pull + restart risuai-nodeonly
```
즉 **deploy 실패 = 이 체인 어느 홉이 끊겼다**. 원인이 AWS Lightsail API 쪽이 아니라 대부분 VM 안 docker compose 상태다.

### 체인 점검 순서
1. `gh run view <id> --log-failed` → 어느 단계에서 실패?
2. CF 에러 코드:
   - `1033` = public hostname에 붙은 tunnel connector 등록 없음 (앱 tunnel cloudflared 죽음)
   - `530` = origin unreachable (tunnel은 붙었는데 내부 서비스가 응답 안 함)
3. VM ssh 들어가 `docker ps`, `docker logs <컨테이너> --tail`

## 8. docker-compose.yml 이미지 필드 실수

**이번 버그**: `cloudflared`, `cloudflared-deploy`, `watchtower` 서비스 세 개가 전부 `ghcr.io/blissful-y0/rsnd-risuai-nodeonly:latest`(앱 이미지)로 설정돼 있었음. 컨테이너는 `tunnel --token ...` 명령을 실행하려 했지만 이미지는 Node 앱이라 `Cannot find module '/app/tunnel'`로 crashloop. 결과적으로 CF 1033/530 동시 발생.

**교훈**:
- compose 파일 수정 후에는 `grep -E '^  [a-z].*:$|image:' docker-compose.yml`로 **서비스↔이미지 매핑을 일람**한다.
- 템플릿(`deploy/lightsail/docker-compose.separate-deploy-tunnel.example.yml`)을 source of truth로 둔다. 손으로 수정한 compose랑 drift가 생기면 이런 사고가 난다. 가능하면 VM에 템플릿을 그대로 symlink/copy하고 `.env`로만 차이를 둔다.

## 9. docker compose vs docker-compose

Amazon Linux 2023 기본 docker에는 compose 플러그인이 없을 수 있음. 이 VM은 하이픈 버전(`docker-compose`)만 있어서 `docker compose up -d`는 `unknown shorthand flag: 'd' in -d`로 실패.

**교훈**: 리모트에서 compose 쓰기 전에 `which docker-compose`, `docker compose version` 둘 다 확인. 스크립트/문서에서는 둘 다 시도하는 fallback을 넣는 편이 안전.

## 10. 토큰 노출되면 즉시 로테이션

**사고 가능성**: 유저가 `docker inspect` 결과를 그대로 복붙하면서 CF Tunnel 토큰 두 개가 채팅에 평문으로 노출됨.

**교훈**:
- 어떤 채널이든 secret이 노출되면 "바꿀지 말지"가 아니라 **즉시 refresh**가 기본. Cloudflare Zero Trust → Tunnels → 해당 tunnel → Refresh token.
- 문서/스니펫 공유 시 `docker inspect`보다 `docker inspect --format`로 **필요한 필드만** 뽑게 유도한다.

## 11. GH Actions 부분 재실행

**쓸 만한 패턴**:
```bash
gh run rerun --repo <owner>/<repo> <run-id> --failed
```
빌드(오래 걸림)까지 다시 돌리지 않고 실패한 deploy job만 재시도. 이미지는 이미 GHCR에 올라가 있으니 웹훅만 다시 쏜다.

## 12. 배포 후 "클라이언트 캐시" 의심하기

**사고 흐름**: 유저가 "4.7 선택했는데 xhigh 안 보임" → 당황해서 watchtower 갱신 여부부터 팠다.

**실제 원인**: 이미 번들에는 들어갔고 (`grep xhigh /app/dist/assets/*.js` hit), index.html이 브라우저/CF edge에 캐시돼 구 번들을 로드하고 있었음. Hard refresh로 해결.

**교훈**: "배포된 이미지 버전 확인(`/app/package.json`)"과 "번들 안에 새 문자열 있나(`grep <새 flag명> /app/dist/assets/*.js`)" 두 개를 먼저 보고, **들어가 있으면 클라이언트/엣지 캐시 의심**하는 순서. Cloudflare 설정에서 `index.html`은 적극 캐시 안 되게 하거나 cache-busting 확인 필요.

## 13. 브랜치/워크트리 정리 원칙

- merged → unmerged 순
- 각 단계 전 현황 dump해서 유저 컨펌
- 워크트리 있는 브랜치는 `git worktree remove`가 먼저
- remote 삭제는 로컬 삭제 이후
- `release/*`는 태그 찍고 지워도 됨 (태그가 영속적 포인터)

## 14. 유사시 재진입 키

Lightsail 임시 SSH 접근법(로컬 pem이 없을 때):
```bash
AWS_PROFILE=risuai aws lightsail get-instance-access-details \
  --instance-name risuai-cloud --protocol ssh --output json
# .accessDetails.privateKey + .certKey + .hostKeys 로 one-shot ssh
ssh -i <id> -o CertificateFile=<cert> -o UserKnownHostsFile=<known_hosts> ec2-user@3.35.106.142
```
키는 1시간 내외 만료. 작업 끝나면 임시 파일 삭제.

## 15. 대시보드 보다 런타임 증거

"Lightsail 콘솔 상태 = running" / "GH Actions 체크 = 녹색"만 믿지 말기. 실제 판단은:
- `docker ps` (컨테이너 상태/이미지 태그)
- `docker logs <svc>` (에러 메시지)
- `docker inspect <svc> --format '{{.Image}} {{.Config.Cmd}}'` (실제 구성)
- `curl https://<domain>/` (end-to-end 응답 코드)

여기서 일치하지 않는 부분이 곧 버그.
